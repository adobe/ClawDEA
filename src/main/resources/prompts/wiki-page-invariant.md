# Invariant-first wiki page template

Use this structure for `pipeline` and `runtime-behavior` concepts. These are
pages that must prevent a reasoner from drawing wrong conclusions about how
the system behaves at runtime, not just help a reader locate code.

Produce the page with exactly these sections in this order:

```markdown
# <concept name>

**Purpose** (one sentence stating what this subsystem does).

## Invariants

- <bullet: a sentence that must be true for reasoning to produce correct results>
- <3–7 bullets total; each bullet cites the code file or spec that makes it true,
  using a standard Markdown link like `[ClassName](../../../src/...)` relative to
  the wiki page location (concept pages live three levels deep)>
- <invariants are statements a caller can rely on, not implementation notes>

## Resolution pipeline

<ASCII, numbered list, or short prose trace of the steps from input to observable
 output. Call out explicitly:
 - cache boundaries
 - registration or binding order
 - which thread/context each step runs on
 - where failures short-circuit
 Keep this to ~10–20 lines.>

## Anti-patterns

- **<name>** — <one-line description>. Triggered when <condition>. Observed in
  <PR/ticket if any>.
- <include at least one named anti-pattern the reasoner should watch for>

## Source pointers

<auto-generated section; humans do not hand-edit this block>

- [ClassName](../../../src/main/kotlin/...) — one-line role description
- [OtherClass](../../../src/main/kotlin/...) — one-line role description
```

## Writing rules

- **Invariants are the load-bearing section.** Every invariant must be the
  kind of statement a reasoner can cite and refuse to violate ("a policy bound
  at `page/v2/page` does not affect pages whose template uses `page/v3/page`").
- **Not navigation.** Do not write "see ClassName for details." Instead, state
  what ClassName guarantees and cite it.
- **Resolution pipeline reflects runtime order**, not file layout. If a
  listener registers late and processes events from a queue, say so.
- **Anti-patterns name traps** with conditions and observed incidents so the
  reasoner recognizes the shape of the bug in a diff.
- **Source pointers** use standard Markdown links with paths relative to the
  wiki page. Do NOT use the `{[ref:...|...]}` chat-only syntax in markdown
  files. Do NOT create new `[[concept]]` wikilinks.
