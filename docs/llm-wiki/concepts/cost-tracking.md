# Cost tracking and session model

**Purpose** Account for per-turn spend, surface it as a running session/daily total in the toolbar, and capture the real session model for display. Cost is the CLI's reported `total_cost_usd` when present, else a **notional cost computed from token usage** via a local price table (subscription/Bedrock plans report `0`). Both the cost figure and the captured model are **accounting/display only** — neither ever feeds back into how the next CLI process is launched.

## Invariants

- **The captured session model is never used to select the model.** `EventStreamHandler.currentModel` is *seeded* from `CliEvent.SystemInit.model`, then *refreshed* from each top-level `CliEvent.AssistantMessage.model` (parsed by `CliEventParser` from the nested `message.model`). The assistant-message value is the authoritative source for the footer because `SystemInit.model` is blank on resume and absent in some CC init shapes; `currentModel` is used only for the per-turn footer and the `perModelUsd` accounting key. The model actually passed to the CLI is resolved independently in `CliProcess` from `ClawDEASettings.getCliModelId(...)` → `buildModelArg` → `--model`. There is no path from `currentModel` (or the resumed transcript) to `--model` ([EventStreamHandler.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/EventStreamHandler.kt), [CliProcess.kt](../../../src/main/kotlin/com/adobe/clawdea/cli/CliProcess.kt)).
- **Per-turn cost is summed, not overwritten — and falls back to a token-priced figure.** `recordTurn` computes the *effective* turn cost via `effectiveTurnCost`: the CLI's `total_cost_usd` (`CliEvent.Result.costUsd`, parsed from the end-of-turn field) when it is `> 0`, otherwise `ModelPricing.costFor(model, …)` over the turn's `usage` token breakdown. The effective figure is *summed* into `sessionUsd`, `perModelUsd`, and the daily total. (Contrast with `contextTokens`, which `EventStreamHandler` overwrites because it is already a per-turn context-size figure — see [CLI bridge](cli-bridge.md).) ([CliEventParser.kt](../../../src/main/kotlin/com/adobe/clawdea/cli/CliEventParser.kt), [CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ModelPricing.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ModelPricing.kt)).
- **The token-priced fallback exists because subscription/Bedrock plans report `total_cost_usd = 0`.** `CliEvent.Result` carries the per-turn `inputTokens`/`outputTokens`/`cacheReadTokens`/`cacheCreationTokens` from the result `usage` object precisely so the chip keeps moving on flat-rate plans. `ModelPricing` holds published per-million-token rates (longest-prefix model match, conservative non-zero fallback for unknown ids; cache-read billed at 0.1× input, cache-creation at 1.25×). `recordTurn`'s `contextTokens` parameter remains intentionally unused (`@Suppress("UNUSED_PARAMETER")`) — context size is not a cost input ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ModelPricing.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ModelPricing.kt)).
- **Session total is per-project, shared across chat tabs.** `CostTracker` is a `@Service(PROJECT)` with a single `sessionUsd`. Opening a second chat tab in the same project shares the counter. The daily total is broader still — it lives on the **application-level** `ClawDEASettings` and accumulates across all projects ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt)).
- **The daily total rolls over by date, not by clock.** `rolledDaily` resets the accumulator whenever `dailyCostDate != today`; there is no timer. A turn on a new day is the trigger. The read-modify-write on the shared `ClawDEASettings` is wrapped in `synchronized(settings)` so concurrent projects don't lose updates ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt)).
- **Resume reseeds the session from the transcript and returns the figure for a footer.** `seedFromResume` calls `resetSession()`, then reconstructs the prior conversation's cost from its `.jsonl` via `TranscriptCostReader.readResumeCost`, seeds the total as a baseline (`seedSession`), and **returns a `TranscriptCostReader.ResumeCost(totalUsd, lastModel)`**. `SessionManager.resumeSession` uses that return value: when `totalUsd > 0` it renders a per-turn-style cost footer (`renderer.renderCostInfo(lastModel, effort = null, totalUsd, elapsed = 0)`) so a reopened conversation shows its accumulated cost at the bottom of the chat, not just on the toolbar chip. This restores the running session figure after a resume; it does **not** touch the daily total (which already accumulated when those turns originally ran) ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [SessionManager.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/SessionManager.kt)).
- **Transcript reading prices token usage and de-duplicates streamed copies by message id.** `TranscriptCostReader.readResumeCost` (which `sumCost` now delegates to) calls `priceLine(line, lineIndex, seen)` on every line: lines without a `"usage"` object (or without a `"model"`) return `null` and contribute `0`; the rest are priced through `ModelPricing.costFor` over their `input`/`output`/`cache_read`/`cache_creation` token counts. Crucially, Claude Code rewrites each assistant message to the `.jsonl` multiple times while streaming (observed up to ~4 byte-identical copies sharing one `message.id`), so `priceLine` keys each line on its `message.id` (falling back to a per-line key when absent) and a **repeat id is priced once** — a duplicate returns `0.0` cost but still reports its model, so the footer reflects the latest turn. Without this de-dup, resume cost over-counted by ~3×. This mirrors the live `effectiveTurnCost` fallback so a resumed session reconstructs the same notional total. Extraction is hand-rolled string scanning (no JSON parse); any exception swallows to `0.0` because seeding is non-critical. The transcript path scheme (`~/.claude/projects/-<encoded-base-path>/<sessionId>.jsonl`) mirrors `SessionScanner` and must stay in sync with it ([TranscriptCostReader.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/TranscriptCostReader.kt), [ModelPricing.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ModelPricing.kt)).
- **The warning band is provider-dependent.** When the effective provider is `subscription` and a `SubscriptionWindow` is present, the band comes from `bandForWindow` (max of the 5-hour / 7-day rate-limit percentages). Otherwise it comes from `bandForDollars` (daily spend vs the user's daily budget). A daily budget of `0` means "no budget" → `NEUTRAL` ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt)).

## Resolution pipeline

1. The CLI emits a `result` event at end of turn. `CliEventParser` extracts `total_cost_usd` into `CliEvent.Result.costUsd` and the `usage` token breakdown into `inputTokens`/`outputTokens`/`cacheReadTokens`/`cacheCreationTokens`.
2. `EventStreamHandler` (on `CliEvent.Result`) renders the per-turn footer (`renderCostInfo(model, effort, costUsd, elapsed)`) using `currentModel` for the model label, then calls `CostTracker.recordTurn(currentModel, event.costUsd, event.contextTokens, event.inputTokens, event.outputTokens, event.cacheReadTokens, event.cacheCreationTokens)`.
3. `recordTurn` computes the effective cost (`effectiveTurnCost`: reported `total_cost_usd` if `> 0`, else `ModelPricing.costFor` over the tokens) and, when `> 0`, adds it to `sessionUsd`, merges into `perModelUsd` keyed by model, and folds into the application-level daily total, then `publish()`es.
4. `publish()` builds a `CostSnapshot` (provider id, session, daily, budget, band, per-model map, window) and fires it on the `CostSnapshotListener.TOPIC` message bus **synchronously while the instance lock is held** — listeners must be quick and must not call back into `CostTracker`.
5. `CostChip` (toolbar `JBLabel`) receives the snapshot, hops to the EDT via `invokeLater`, and re-renders label text (provider-aware via `formatText`), color (`bandColor`), and tooltip. Clicking the chip prompts for a daily budget.
6. On resume, `SessionManager.resumeSession` calls `CostTracker.seedFromResume(sessionId)` *before* restarting the bridge, so the chip shows the restored session total immediately. The returned `ResumeCost` (de-duplicated total + last model) also drives a one-off `renderCostInfo` footer appended to the chat history (`effort = null`, `elapsed = 0`), mirroring the live per-turn footer.

## Anti-patterns

- **Feeding the captured/display model back into `--model`** — `currentModel` and the resumed transcript are accounting/display inputs only. The launch model is resolved from settings; do not wire `SystemInit.model` into command assembly or a resume would silently pin the model.
- **Treating `total_cost_usd` as cumulative, or token-pricing when a real dollar figure exists** — `total_cost_usd` is per-turn; `recordTurn` sums turns. `ModelPricing` is a *fallback* used only when the reported cost is `0` (flat-rate plans); never recompute from tokens when the CLI already reported a non-zero dollar figure, or you'll diverge from the authoritative number.
- **Reseeding the daily total on resume** — `seedFromResume` only reseeds the session view. The daily total already accumulated when those turns ran; reseeding it would double-count.
- **Pricing every transcript line** — Claude Code writes each assistant message to the `.jsonl` several times while streaming (same `message.id`, identical `usage`). Summing every line over-counts resume cost by ~3×. De-duplicate by `message.id` (the `seen` set in `priceLine`) and price each logical turn once.
- **Heavy or re-entrant `CostSnapshotListener` work** — listeners run synchronously under the `CostTracker` lock. Block or call back in and you deadlock or stall the turn. Marshal to the EDT and return fast (as `CostChip` does).
- **Treating the session total as per-tab or per-conversation** — it is per-project and shared across tabs; the daily total is application-wide. Don't assume isolation.
